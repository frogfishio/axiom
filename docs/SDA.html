Structured Data Algebra

SDA: What It Is, Why It Exists, and How It Works

A detailed introduction for mathematically and computationally literate readers who care about semantics, determinism, and the recovery of trustworthy meaning from untrustworthy structured data.

Abstract. SDA is a small, pure language for structured-data transformation. Its aim is not mere convenience, nor the imitation of existing query tools, but the establishment of a stable semantic kernel for extraction, normalization, validation, and reduction over tree-shaped data. The central claim is that much ordinary data work fails not because transformation is impossible, but because the semantics of transformation are left underspecified. SDA exists to remove that underspecification.

The Problem SDA Is Trying to Solve

Modern software is saturated with structured data and saturated equally with semantic ambiguity. JSON payloads, event envelopes, API responses, message buses, configuration trees, audit records, and telemetry snapshots all present themselves as if their structure were already meaningful. It rarely is.

The difficulty is not merely that data may be missing or malformed. The deeper difficulty is that ordinary host-language access paths leave the status of data unclear. A path expression in JavaScript, Python, or even a typed language often confounds several quite distinct questions:

  • Was the field present?
  • Was the container of the right shape?
  • Was there one answer, no answers, or many answers?
  • Was a default silently introduced?
  • Was a duplicate collapsed?
  • Was an operational accident reinterpreted as a semantic fact?
The usual problem is not that a program cannot reach data. It is that, having reached something, it cannot say with precision what that something means.

SDA begins from the claim that this is not a peripheral inconvenience. It is the central defect of ad hoc structured-data transformation. If a data language cannot distinguish absence from null, multiplicity from uniqueness, and wrong shape from missing value, then it cannot support dependable reasoning.

What SDA Is

SDA stands for Structured Data Algebra. It is a small, formal, pure language for transforming structured data without hidden coercions and without surrendering key semantic distinctions.

What it is

  • a transformation language
  • a semantic kernel
  • a precise model of extraction, normalization, and reduction
  • a tool that can be embedded or run standalone

What it is not

  • not a general scripting runtime
  • not a replacement for host programming in general
  • not a bag of JSON tricks
  • not a workflow engine

One may put the matter plainly: SDA exists only if it yields stronger semantic guarantees than ordinary host-language data munging. Its purpose is certainty.

The Mathematical Shape of the Language

At a high level, an SDA program is a pure transformation from a host-supplied input value to a resulting value or an explicit failure condition. One may describe the denotation schematically as:

$$\left[\!\left[ e \right]\!\right] : Env \times V \to V \cup F$$

Here:

  • $V$ is the space of SDA values
  • $Env$ is the lexical environment for names
  • $F$ is the family of explicit failure values

The important point is not the notation itself, but what it excludes. There is no ambient I/O in the core meaning. There is no hidden state. There is no host callback smuggled into evaluation. There is no “best effort” reinterpretation that silently mutates the semantic question.

Nor is SDA merely a pointwise map over ordinary JSON values. It distinguishes several carriers because those distinctions matter:

$$V ::= Null \mid Bool \mid Num \mid Str \mid Seq(V) \mid Set(V) \mid Bag(V) \mid Map(Str, V) \mid Prod(Label, V) \mid \cdots$$

Why keep these distinctions? Because different carriers tell different semantic truths:

  • a sequence says order matters
  • a set says uniqueness matters
  • a bag says multiplicity matters
  • a map says lookup is by key
  • a product says field structure is part of the meaning

A language that erases these too early also erases the very evidence on which later reasoning depends.

Core Ideas

1. Absence is not null

One of SDA's first commitments is that the absence of a key and the presence of a key whose value is null are not the same condition. Many practical systems collapse these. SDA refuses that collapse because it destroys information.

2. Multiplicity is not a nuisance

Duplicate-bearing structures occur constantly in the real world: repeated headers, repeated observations, repeated records, repeated candidate matches. Most everyday tooling either ignores or prematurely normalizes them. SDA keeps multiplicity explicit until the user asks for normalization.

3. Failure is semantic, not merely operational

In SDA, failure is not just “the interpreter got upset.” A missing required key, a wrong-shape access, or an illegal normalization condition is part of the defined semantic space. This is why the tool emits named failure values rather than vague host-language accidents.

4. Determinism matters

If two conforming implementations of the language can disagree because of host map ordering, incidental parser behavior, or library quirks, then the language is not yet well defined. SDA therefore treats determinism as part of the language contract, not a performance detail.

5. Surface convenience must answer to the core

Sugar is acceptable only when its meaning is precise. The point of SDA is not to produce pretty syntax detached from semantic law. The point is to ensure that every surface form desugars to a stable underlying meaning.

How SDA Looks in Practice

The surface language is intentionally small. Here are a few representative patterns.

Required projection

input<"name">!

This asks for a required `name` entry from the incoming value. If the key is missing, the result is not a vague host error but a defined SDA failure.

Optional projection

input<"name">?

This asks the softer question: is there such a key, and if so what is its value?

Value extraction over a JSON object

values(input)

Filtering a sequence

{ x in input | x > 2 }

Conceptually: range over the incoming sequence, keep those elements satisfying the predicate, and preserve the carrier.

Projection under a filter

{ yield x * 10 | x in input | x > 2 }

Piped composition

input |> values(_) |> count(2, _)

The pipe is explicit. The placeholder is explicit. The language does not pretend that every right-hand side is an implicit host-language function application.

Rebuilding structure under explicit success flow

bindRes(
  input<"name">!,
  name => bindRes(
    input<"city">!,
    city => Ok(Map{"person" -> name, "location" -> city})
  )
)

What is notable here is not complexity for its own sake. It is that required extraction remains visibly required all the way through the transformation.

A Small Piecewise Example

One can think of required keyed access in the standalone profile as obeying a case distinction of the following general sort:

$$ select_{!}(x, k) = \begin{cases} Ok(v) & \text{if } x \text{ has exactly one value at key } k \\ Fail(t\_sda\_missing\_key, \text{"missing key"}) & \text{if no such value exists} \\ Fail(t\_sda\_duplicate\_key, \text{"duplicate key"}) & \text{if strict keyed multiplicity is violated} \\ Fail(t\_sda\_wrong\_shape, \text{"wrong shape"}) & \text{if } x \text{ is not a lawful keyed carrier} \end{cases} $$

The point is not that every user must think in piecewise functions while using the tool. The point is that a rigorous language ought to admit such a description. If it does not, it is usually because the semantics are being improvised rather than specified.

Where SDA Sits in the Larger System

SDA is not the whole Axiom family. It is the reduction kernel. Its role is to reduce noisy structured input into semantically sharper artefacts. In the larger architecture:

  • SDA reduces
  • ENR expands relationally
  • ENR+ ranks, explains, and resolves enriched candidates
  • Axiom orchestrates effectful movement between stages

Why isolate reduction in this way? Because data workflows typically alternate between phases of semantic sharpening and phases of semantic expansion. SDA owns the sharpening phase.

$$ raw\ input \; \xrightarrow{SDA} \; artefact\ A \; \xrightarrow{ENR} \; relational\ artefact\ E \; \xrightarrow{SDA} \; artefact\ B $$

This alternation is architectural, not cosmetic. If extraction, validation, and normalization are not kept semantically honest at the reduction stage, everything built on top becomes less trustworthy.

Why This Matters Now

Contemporary systems increasingly rely on data assembled from partial observations, loosely governed interfaces, and operationally noisy boundaries. In such environments the central intellectual problem is not “how to parse JSON.” It is how to recover dependable meaning from unreliable evidence.

That is why a language like SDA is worth taking seriously. Its ambition is modest in one sense, because it is deliberately small. Yet it is ambitious in the deeper sense, because it insists that the semantics of structured-data work ought to be explicit, deterministic, and testable.

The intellectual wager behind SDA is that structure alone is not enough. What matters is whether the transformation language preserves the semantic facts on which one wishes to reason.

From that point of view, SDA is best understood neither as a scripting convenience nor as a fashionable domain-specific language. It is a proposal about where precision ought to reside in software systems that live by structured data.
